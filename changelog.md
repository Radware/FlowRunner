## Unreleased — Inline step inspector: Basic/Power progressive disclosure (UX Wave 2, lane inspector)
- **Decision:** evolved `createStepEditor` (in `flowStepComponents.js`) into a persistent inline **inspector** with a **Basic/Power** segmented toggle in the editor header. **Basic** keeps the author in context — common fields (name, method, URL) plus an at-a-glance **summary chip row** (`buildInspectorSummary`) that distills each step type (request: method/URL/headers/extracts/body; condition: if/then/else; loop: source/var/body; transform: op & output counts). **Power** reveals the advanced blocks (the Headers/Body/Extract/Options tab strip) by toggling `inspector-mode-basic`↔`inspector-mode-power` classes on the editor root; advanced blocks are marked `.power-only` and hidden via CSS in Basic mode. **Why:** the old editor dumped every field at once; progressive disclosure lowers cognitive load for the common case while keeping power fully reachable.
- **Disclosure is view-only:** switching Basic↔Power **never** marks the editor dirty and never touches `localStep` — save/cancel/dirty behavior is unchanged (regression-tested). The summary refreshes on save and on cancel-revert so it always reflects persisted values.
- **Read-only raw `##VAR##` marker preview:** the Body tab now surfaces a **read-only** textarea (`.raw-body-markers`, `readonly`+`tabindex=-1`) showing exactly what gets persisted — `preProcessBody(localStep.body)` output — so authors can see the `{{var}}`→`##VAR:string|unquoted:name##` conversion. **Critical contract:** this view is display-only and is **never read back**; a hand-typed `{{var}}` in it cannot reach saved body JSON (the body still routes through `preProcessBody` unchanged from the editable textarea above). Guarded by a test that types a hostile `{{EVIL}}` into the raw view and asserts it never appears in the saved step.
- **Tokens/CSS:** appended a delimited `/* === WAVE2 LANE inspector === */` block to the END of `styles.css`, consuming only OKLCH semantic tokens (`--surface-sunken`, `--accent`, `--accent-contrast`, `--border`, `--border-strong`, `--text`, `--text-muted`, `--focus-ring`) — no hardcoded colors, no `#000`/`#fff`.
- **Scope/ownership:** touched only `flowStepComponents.js` + the appended `styles.css` block. Did **not** edit `flowVisualizer.js` (canvas lane owns it); the node-graph modal replacement is deferred to the engine work. No new modules, no deps, no `build.files` change (both edited modules already listed). TDD: `__tests__/flowStepInspector.test.js` (9 tests). Suite: 253 → **262 passing**.

## Unreleased — Per-request retries/backoff + cURL/HAR import (UX Wave 2, lane engine-features)
- **Decision (retries):** `_executeRequestStep` in `flowRunner.js` now honors an **additive** request-step field `step.retries = { count, delayMs }`. A retry fires on a **non-2xx HTTP status OR a network/fetch error**, up to `count` times, sleeping `delayMs` (via the existing `_sleep`, which short-circuits on `stopRequested`) between attempts. Each attempt gets a **fresh `AbortController` + 30s timeout** (an aborted/timed-out controller can't be reused). **Why:** transient upstream failures are common in live demos; a bounded retry avoids a whole-flow stop on a blip. Default is `count: 0`, which is **byte-for-byte the previous single-attempt behavior** — the `onFailure` (`stop`/`continue`) semantics are unchanged and only apply once retries are exhausted. A **user-requested stop/abort is never retried**. CROSS-REPO FOLLOW-UP: `step.retries` is additive in `.flow.json`; flowrunner-cli (Python) must eventually honor it (until then it's ignored, which is safe).
- **Decision (import):** new **zero-dependency** module `importCurlHar.js` — `parseCurl(str)` → one request step, `parseHar(objOrJson)` → request steps (one per http(s) entry). Both emit the canonical `createNewStep('request')` shape (the inverse of `harExporter.js`). The cURL path uses a small shell-style tokenizer (single/double quotes, backslash escapes, `\`-newline continuations) and maps `-X/--request`, `-H/--header`, `-d/--data*`, `--url`; a data flag implies POST unless a method is given; unknown boolean flags (`--compressed`, `-k`, …) are ignored. HAR skips non-http(s) entries and HTTP/2 pseudo-headers. **Why zero-dep:** CSP `script-src 'self'` + the build.files discipline make vendored parsers costly; the surface we need is small.
- **UI hook:** `importCurlHarUI.js` wires two sidebar buttons (`#import-curl-btn` via `prompt`, `#import-har-btn` via a hidden file input) that append parsed steps to the current flow and mark it dirty. index.html edit is additive (three nodes in `.sidebar-actions`, reusing existing `.btn` classes — no new CSS/tokens needed). Wired in `app.js` after `initializeEventListeners()`.
- **Deps + build.files:** no new deps. Added `importCurlHar.js` + `importCurlHarUI.js` to `build.files` (the #1 rule).
- **Tests:** TDD — `__tests__/flowRunner.test.js` +8 retry cases (mocked fetch), new `__tests__/importCurlHar.test.js` (20 parse cases). Suite: 253 → **281 passing**.

## Unreleased — OKLCH design-token foundation + projector theme (UX Wave 1, lane ux-p1)
- **Decision:** introduced a two-tier OKLCH token layer in `styles.css` — a primitive `--ramp-*` ramp feeding semantic tokens (`--surface`, `--surface-raised`, `--text`, `--text-muted`, `--accent`, `--border`, `--run-success|error|skipped|running`, …). **Why:** unblocks the whole UX overhaul with one place to define color, and OKLCH's perceptual uniformity keeps contrast honest across the light/dark flip. The **default theme is LIGHT, tuned for projector legibility** (the core demo scenario: an SE mirrors a live API attack/defense flow to a projector in a lit room). Dark is opt-in via `[data-theme="dark"]` and `@media (prefers-color-scheme: dark)` (the latter suppressed by an explicit `data-theme="light"`). Neutrals are tinted toward the brand hue (262°) so they never land on pure `#000`/`#fff`.
- **Removed** the gradient wordmark (`background-clip:text` → a single solid `var(--accent)`) and the `.app-branding` `linear-gradient`. **Why:** clipped-text gradients render soft and low-contrast on projectors.
- **Proof surfaces only** (not a full restyle — that's Wave 2): workspace header, primary button, branding wordmark, and a new token-driven run-status **pip** (a solid dot before `.result-status` badges) now consume semantic tokens. Legacy `--primary-color` etc. remain until Wave 2 migrates the rest.
- Contract documented in `docs/design-tokens.md`; guarded by `__tests__/designTokens.test.js` (11 tests). No JS behavior change, no new deps, no `build.files` change.

## Unreleased — Persistent recent-files + sidecar workspace model (data layer)
- **Decision:** added `electron-store` (main process) for durable, cross-session persistence of the **recent-files** list (there was NO persistence before — recents lived only in renderer `localStorage`, lost on a cleared cache). New `store:*` IPC (`getRecentFiles`/`addRecentFile`/`removeRecentFile`/`setRecentFiles`/`clearRecentFiles`) in `main.js`, bridged in `preload.js`. `fileOperations.js` keeps `localStorage` as the fast synchronous mirror (so every existing sync caller is untouched) and mirrors each mutation to the durable store; `hydrateRecentFiles()` reconciles the store back on startup (store wins; if empty, it's seeded from `localStorage`). Store keys/name live in `config.js`.
- **Decision:** organization metadata (folders/tags/category) is persisted in a **sidecar** `.flowrunner/workspace.json` under the app's userData dir — never inside `.flow.json`. **Why:** additive org fields on the flow file are *lossy through FlowRunner's own reader* (`jsonToFlowModel` drops unknown keys on re-save) and would violate the additive-only cross-app contract; the sidecar keeps every `.flow.json` byte-clean and git-diffable. New pure module `workspaceManager.js` (no Electron/`fs` imports, so it unit-tests under jsdom) owns the model: a `parent_id` folder tree plus per-flow `{ path, folderId, tags[], category }`, loosely aligned with the ShowRunner portal (UUID folder ids, `tags[]`, `category`) for future interop. `workspace:load`/`workspace:save` IPC read/write the sidecar; the renderer owns mutations. TDD: `__tests__/workspaceManager.test.js` (47 tests). Added `workspaceManager.js` to `build.files`.
- **Scope:** DATA/persistence only — no sidebar tree UI (Wave 2), no `.flow.json` format change.

## Unreleased — Auto-layout foundation (spike + engine-agnostic adapter)
- **Decision (spike):** ship **elkjs** for flow-graph auto-layout via its **bundled build run on the main thread** (`import ELK from 'elkjs/lib/elk.bundled.js'`, no `workerUrl`). **Why:** FlowRunner's renderer CSP is `script-src 'self'` with **no `worker-src`** (falls back to `script-src`) and **no Node integration** (`contextIsolation:true`), so a `blob:`-URL Web Worker (how most bundlers ship ELK) would be **blocked**, and the Node `web-worker` path is unavailable. The bundled build's inlined **synchronous "fake worker"** touches none of the CSP-restricted primitives (`Worker`/`Blob`/`importScripts`) — the only unconditionally CSP-clean path without relaxing the policy. Flows are small (tens of nodes), so main-thread layout is single-digit-ms; a real worker buys nothing and would need a same-origin worker file plumbed through `build.files`. Full findings + the future same-origin-worker escape hatch in [docs/auto-layout-spike.md](docs/auto-layout-spike.md).
- **New module `autoLayout.js`:** engine-agnostic adapter, **not yet wired into `flowVisualizer.js`** (Wave 2 adds a "Tidy Up" button). `computeLayout(steps, { engine, nodeSizes, direction })` → `{ positions: { [stepId]: {x,y} }, engine }`. Input respects then/else/loop nesting and accepts **both** the in-memory model shape (`thenSteps`/`elseSteps`/`loopSteps`) and the on-disk `.flow.json` shape (`then`/`else`/`steps`). Uses ELK **layered + `hierarchyHandling:INCLUDE_CHILDREN` + orthogonal edge routing**; output is flat, top-left, origin-normalised, **deterministic** (stable document-order feed + fixed spacing).
- **Fallback engine:** **`@dagrejs/dagre`** (pure-JS, synchronous, equally CSP-clean). Used when ELK throws or when `engine:'dagre'` is requested; result reports which engine actually ran. Dagre reports node centers — adapter converts to top-left.
- **Deps + build.files:** added `elkjs` and `@dagrejs/dagre` as `dependencies`; added `autoLayout.js` to `build.files` now (per the #1 rule) so it's packaged the moment Wave 2 imports it.
- **Tests:** `__tests__/autoLayout.test.js` — 21 TDD cases (graph-in → positions-out; linear/branches/nested loop+condition; both field-name shapes; determinism; node-size respect; engine fallback) run against **both** engines. Suite: 126 → **147 passing**.

## Unreleased — CI: Windows release ships the NSIS installer
- **Decision:** the Windows asset (`FlowRunnerSetup-x64-win-{VERSION}.zip`) now contains the electron-builder **NSIS installer** (`FlowRunner Setup {VERSION}.exe`) instead of the zipped `win-unpacked` portable directory. **Why:** `win.target` was already `nsis`, so the installer was being built and then discarded; shipping it gives recipients a real install experience (choose dir, Start-Menu shortcut, uninstaller) — better for handing demo builds to SEs/customers.
- **Fix:** the first CI attempt failed (`zip I/O error`, exit 15) — the zip step wrote to `../release` while two levels deep in `artifacts/windows-latest-build`; corrected to `../../release`. → [gotchas.md](gotchas.md), Build & packaging.
- Shipped as an update to the existing **v1.2.1** release (no version bump).
- **CI trigger overhaul (`build.yml`):** added `paths-ignore` (`**.md`, `docs/**`, `.vscode/**`) so docs-only pushes no longer rebuild; added `workflow_dispatch` (manual, any branch) and `pull_request` triggers so branches build on all three platforms **without publishing**; gated the `release` job to `push` on `main`/`master`. **Why:** CI previously built only on push-to-main with no way to validate a build off `main`, blocking safe testing of dependency upgrades (e.g. the Electron bump). Verified: a `push` run publishes; a `workflow_dispatch` run builds all platforms with `release` **skipped**.
- **CI:** added `schemas/**` to `paths-ignore` so schema-doc changes don't trigger a rebuild.
- **Fix (schema drift):** `schemas/flow-v1.schema.json` declared the internal model names `thenSteps`/`elseSteps`/`loopSteps`; corrected to the real wire keys `then`/`else`/`steps` (verified against sample flows and `flowCore.js:111-116`; the CLI reads only these).
- **Research / proposal:** produced the multi-agent councilled [FlowMap & UX evolution proposal](docs/flowmap-evolution.md) and documented the **cross-app FlowMap contract** (three apps share `.flow.json`: FlowRunner UI, flowrunner-cli, ShowRunner portal) in `architecture.md`/`gotchas.md`/`masterplan.md` + the schema `$comment`. **Direction:** sequence over big-bang; graceful-degradation (unknown step/op ⇒ skip-with-warning, never crash) first; adopt React Flow for the node view only behind packaged-build evidence; `schemaVersion` + subflows gated on *deployed* CLI support.
- **Fix (graceful degradation — JS engine, parity with the CLI):** the runner no longer **crashes** on an unknown step type (`flowRunner.js` threw → halted the whole run) and `transformOps.js` no longer **silently downgrades** an unknown transform op to `base64_decode` (the exact bug just fixed in flowrunner-cli). Both now **skip-with-a-machine-readable-warning** and continue: an unknown step type yields a `status:'skipped', unsupported:true` result + a user-visible warning; an unknown transform op is recorded in `executeTransformOps`' `warnings[]` (`TRANSFORM_OP_UNSUPPORTED` marker) and surfaced by the transform step. `normalizeTransformOp` is intentionally left tolerant because the *editor* (`flowStepComponents.js`) calls it — the editor's own silent rewrite of unknown ops on open is a separate follow-up. This is schema-evolution P1 from the proposal.
- **Test harness fix:** `npm test` was fully broken — `__tests__/setup.js` `require()`s the vendored UMD `drawflow.min.js`, which Node treats as ESM under the app's root `"type": "module"`, so **every** suite failed to load (CI never caught it — `build.yml` runs only `npm run dist`). Added `assets/vendor/drawflow/package.json` (`"type": "commonjs"`) scoping just that vendored dir to CJS (browser `<script>` load unaffected). This unblocked the suite (**122 passing**) and revealed **4 pre-existing failures in `flowVisualizer.test.js`** (Drawflow drag/double-click/add-step under jsdom) — pre-dates this change, tracked as a follow-up.

## v1.1.0 — Quality, testing & continuous run *(2025-05)*
- **Refactor:** introduced `state.js` and `uiUtils.js` (centralized state + UI utilities).
- **Testing foundation:** Jest + jsdom and Playwright E2E.
- **Continuous run** mode — **decision:** deliberately scoped down from the roadmap's "background/parallel runner" to a simple "run repeatedly in sequence." The main UI is **not** usable during continuous execution and there's no configurable inter-run delay. Docs/UI references to background/parallel execution were removed to avoid over-promising.
- **Update check** — **decision:** non-intrusive; if GitHub is unreachable, do nothing (no popup). Also available manually via Help → Check for Updates. There is no in-app auto-download — it always links to the GitHub release.
- Also: recent-files removal UI, better validation/error messages, keyboard shortcuts.

## v1.0.0 — Initial release *(2025-04)*
- First public release: visual flow authoring (Request/Condition/Loop), dual views (List/Editor + Node-Graph), variable management (static vars, global headers, extraction, `{{var}}` substitution), local `.flow.json` files, recent-files, run/step/stop engine with real-time highlighting.
- **Decisions:**
  - Built on **Electron** for a self-contained, offline, backend-free desktop tool.
  - Targeted **Windows x64 + macOS arm64** at launch (Linux deferred to v1.2.1).
  - **Primary internal motivation:** help Radware SEs/Architects **demonstrate Business Logic Attack Protection** by authoring and visually presenting both legitimate and malicious API sequences (see [masterplan.md](masterplan.md)).
  - Late hardening: tightened CSP (`script-src 'self'`, hence vendored Drawflow), `shell` support in preload, package restructured for builds.
